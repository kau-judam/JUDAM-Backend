### 작업 2 — 레시피 API 통합 테스트 실행 (로컬 서버 + DB 검증)

**이 작업이 하는 일**: 로컬 서버(`http://localhost:3000`)에서 71개 케이스를 직접 API 요청으로 수행하고, DB에 변화가 생기는 케이스는 AWS RDS에 직접 SQL 쿼리를 실행해 실제 레코드 변화까지 확인했다.

**DB 검증 대상 API (12개)**

| API | 검증 항목 |
|-----|---------|
| 레시피 작성 (TC-01) | `recipes` 테이블 레코드 생성, 초기값 확인 |
| 관심 등록 (TC-10) | `recipe_interests` 레코드 생성, `interest_count` +1 |
| 관심 해제 (TC-14) | `recipe_interests` 레코드 삭제, `interest_count` -1 |
| 댓글 작성 (TC-21) | `recipe_comments` 레코드 생성 |
| 댓글 수정 (TC-25) | `updated_at` 업데이트 |
| 댓글 좋아요 등록 (TC-34) | `recipe_comment_likes` 레코드 생성, `like_count` +1 |
| 댓글 좋아요 취소 (TC-38) | `recipe_comment_likes` 레코드 삭제, `like_count` -1 |
| 대댓글 작성 (TC-57) | `recipe_comments` 레코드 생성, `parent_comment_id` 설정 |
| 댓글 삭제 (TC-33) | 댓글 삭제 + 대댓글·좋아요 cascade 삭제 |
| 양조장 레시피 등록 (TC-42) | `author_type = 'BREWERY'` 확인 |
| 펀딩 전환 (TC-61) | `funding_projects` 레코드 생성, `recipes.status → FUNDING_IN_PROGRESS` |
| 레시피 삭제 (TC-71) | 레시피 삭제 + `recipe_interests`, `recipe_comments` cascade 삭제 |

**펀딩 전환(TC-61) 전제 조건 처리**: `FUNDING_READY` 상태 레시피가 없어 `INTEREST_THRESHOLD`를 100→1로 임시 변경해 테스트용 레시피를 `FUNDING_READY` 상태로 만든 뒤 테스트하고, 즉시 100으로 원상 복구했다.

### 작업 3 — 버그 수정 4건 (`fix/recipe-api-spec-mismatch`)

**이 작업이 하는 일**: 통합 테스트 수행 중 API 명세서와 실제 응답이 다른 4개 케이스를 발견하고 수정했다.

**수정 내역**

| 버그 | 명세서 기대값 | 실제 응답 | 수정 내용 |
|------|------------|---------|---------|
| 댓글 작성 메시지 (TC-21) | `"댓글이 작성되었습니다."` | `"댓글이 등록되었습니다."` | `recipeCommentController.js` 메시지 수정 |
| 댓글 좋아요 중복 (TC-35) | `status:400` / `"이미 좋아요한 댓글입니다."` | `status:409` / `"이미 좋아요를 누른 댓글입니다."` | 상태 코드 409→400 + 메시지 수정 |
| 좋아요 취소 내역없음 (TC-39) | `status:400` | `status:404` | 상태 코드 404→400 수정 |
| 펀딩 날짜 형식 (TC-61) | `"2026-05-20"` | `"2026-05-19T15:00:00.000Z"` | `db.js`에 `pg` DATE 타입 파서 추가 |

**날짜 형식 버그 원인**: ERD상 `funding_projects.start_date` / `end_date`는 `DATE` 타입이지만, `pg` 라이브러리가 이를 JavaScript `Date` 객체로 변환하면서 서버 로컬 시간(KST) 기준 자정이 UTC로 바뀌어 하루 앞당겨지는 문제였다. `types.setTypeParser(1082, (val) => val)`를 추가해 `DATE` 컬럼을 `"YYYY-MM-DD"` 문자열 그대로 반환하도록 수정했다.

### 세부사항

- `app.js`에 머지 충돌 마커(`<<<<<<< HEAD`)가 미해결 상태로 커밋되어 있어 서버가 실행되지 않는 문제가 있었다. 서버를 켜기 위해 임시로 충돌을 해결(`/api/users` 경로 선택)했으나, 이는 팀원이 작성한 코드라 원상 복구했다. 해당 팀원이 직접 충돌을 해결하고 재커밋해야 한다. (`/users` vs `/api/users` 경로 선택, API 명세서 기준으로는 `/api/users`가 맞음)
